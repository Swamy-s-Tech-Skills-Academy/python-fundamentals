## 0) PEP 8: Code Style Warm-Up (~4 min)

You have now written four sessions of Python. Before adding more features, take a moment to write it *well*.

**PEP 8** is the official Python style guide, authored by Guido van Rossum (Python's creator) in 2001. It exists for one reason: **code is read far more often than it is written**.

> Full deep-dive comes in Level 4 (Clean Code). Here you build the habit early.

### Naming conventions

| Rule | Correct | Incorrect |
|---|---|---|
| Use `snake_case` for variables and functions | `total_score` | `TotalScore`, `totalscore` |
| Names should be meaningful | `student_age` | `x`, `val`, `temp` |
| Constants in `UPPER_SNAKE_CASE` | `MAX_SCORE = 100` | `maxScore`, `max_score` |

### Indentation

- Use **4 spaces** per indent level (most editors map the `Tab` key to 4 spaces automatically).
- When a function or `if` block has more than one argument and you wrap to the next line, **align with the opening delimiter or use a hanging indent**:

```python
# One argument — no wrap needed
print("Hello")

# Multi-argument wrap — start from next line with indent
result = some_function(
    argument_one,
    argument_two,
    argument_three,
)
```

**Why it matters:** Python uses indentation to define code blocks — unlike most languages that use `{}`. Wrong indentation causes `IndentationError` and is one of the most common beginner mistakes.

### Spacing rules

```python
# Correct
result = max_score - penalty
scores = [10, 20, 30]

# Incorrect
result=max_score-penalty
scores = [ 10 , 20 , 30 ]
```

- Spaces **around** operators: `x = a + b`, not `x=a+b`
- No spaces inside brackets: `scores[0]`, not `scores[ 0 ]`
- Space **after** commas, not before: `(a, b, c)`, not `(a ,b ,c)`

### Comments

```python
# This is a single-line comment — keep it under 72 characters
total = sum(scores)  # inline comment: explain the why, not the what
```

- Use `#` for comments; Python ignores everything after `#` on that line
- Write comments that explain **why**, not just **what** the code does
- Limit comment lines to ~72 characters

### Why this matters now

By the time you reach the debugging section below, you will see that **clear names and clean spacing** make tracebacks easier to read and bugs easier to spot.

## 1) Syntax vs Runtime Errors

| Error Type | When it appears | Typical examples |
| --- | --- | --- |
| Syntax error | Before program runs | Missing `:`, indentation mistake, unmatched quote |
| Runtime error | During execution | `ZeroDivisionError`, `TypeError`, `IndexError`, `NameError` |

**Reasoning:** If Python cannot parse your code, execution never starts. If parsing succeeds, logic/data issues can still fail at runtime.

## 2) How to Read a Traceback

Use this sequence every time:

1. Start at the **bottom** (actual error type + message).
2. Find file + line number where it failed.
3. Inspect that line and nearby lines.
4. Trace backward to the input/state that produced it.

Common errors for Session 7:

- `NameError` → variable not defined
- `TypeError` → incompatible operation for type
- `ValueError` → invalid value for conversion
- `IndexError` → index outside list bounds
- `KeyError` → dictionary key missing
- `ZeroDivisionError` → divisor is zero

## 3) Common Beginner Mistakes to Watch

- Missing colons in `if/for/while` headers
- Misaligned indentation blocks
- Using `=` when comparison needs `==`
- Comparing string input to integer values without conversion
- Off-by-one assumptions in `range()`
- Forgetting loop updates in `while` loops
- **Compound conditions that “look right” but evaluate wrong** — e.g. mixing `and` / `or` / `not` without parentheses. Python applies **`not` before `and` before `or`** unless you override with `()`. Symptom: loop never runs, runs forever, or branch flips the opposite way — **no traceback**, just wrong behavior.

**Debug tactic:** print the **pieces** of the condition before the `while` / `if`:

```python
ready = count < limit
allowed = not cancelled
print("DEBUG:", ready, allowed, ready and allowed)
while ready and allowed:
    ...
```

For a guided refresher on evaluation order and a tiny compound `while`, run (from repo root): `python src/L1/S6/08_boolean_logic_precedence.py` (see [Session 6](S6.md)).

## 4) Print-Debugging Workflow

When behavior is wrong but script still runs:

1. Print inputs before logic starts.
2. Print values at each decision point.
3. Print loop counters and intermediate totals.
4. Confirm final output against expectation.
5. Remove temporary debug prints after fix.

Use a prefix like `DEBUG:` so cleanup is fast.

## 5) Essential Built-in Functions (Canonical Scope)

| Function | Why it matters |
| --- | --- |
| `len()` | Count items/chars for bounds checks and summaries |
| `max()` | Find the largest value quickly |
| `min()` | Find the smallest value quickly |
| `sum()` | Aggregate totals for reporting/calculations |
| `abs()` | Measure distance from zero (difference magnitude) |
| `round()` | Format numeric output to practical precision |

**Reasoning:** These six built-ins cover the most common beginner data checks and summaries without introducing extra complexity.

## 6) How to Apply Built-ins in Practice

- Combine `sum()` and `len()` for averages.
- Use `max()` and `min()` to detect ranges/outliers.
- Use `abs()` for difference-based comparisons.
- Use `round()` for display-friendly numeric results.

## 7) Troubleshooting Quick Guide

- `SyntaxError` / `IndentationError` → fix structure first
- `NameError` → verify exact variable names
- `TypeError` / `ValueError` → validate and convert input early
- `IndexError` / `KeyError` → check existence before access
- `ZeroDivisionError` → guard divisor before dividing
- **Wrong behavior, no exception** → print sub-expressions in `if` / `while` headers; confirm **`not` / `and` / `or`** order and parentheses (Session 6 optional `08_boolean_logic_precedence.py`)
